# Triaging issues

Triage answers three questions for each issue: what is it, does it belong here, and what happens next. Do it quickly and consistently — a groomed backlog is one where the next thing to work on is obvious.

- **Classify from the repo's own labels.** Type (bug, feature, question, chore) and area. Don't invent a taxonomy the project doesn't use; propose a new label only for a real, recurring gap.
- **Make it actionable or ask.** If you can't tell what "done" means, the issue isn't ready — ask the author one specific question, don't guess. Mark it as needing info.
- **Kill duplicates.** Search before you accept. Link the copy to the canonical issue and close it there, moving any new detail across.
- **Prioritize on evidence.** Weigh impact (how many, how badly) against effort and reversibility — from what the issue actually shows, not a hunch. Reserve the top priority for things that are genuinely burning.
- **Route, don't hoard.** Assign when the right owner is clear; otherwise leave it open with enough context that anyone could pick it up.

Prefer a small, defensible decision to a big reshuffle. When you're unsure, a clarifying question is triage too.
