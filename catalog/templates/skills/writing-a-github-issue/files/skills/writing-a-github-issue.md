# Writing a GitHub issue

An issue is a request for work aimed at someone who wasn't in the room. It succeeds when that person can start without asking you anything.

- **Lead with the problem, not the fix.** Who hurts, when, and what they can't do. A title that names the problem beats one that names a guess at the solution.
- **Make it reproducible or observable.** For a bug: steps, what happened, what you expected, and the environment. For a feature: the situation today and the outcome that would count as solved.
- **State the outcome, hold the implementation.** Say what "done" looks like from the outside. Leave how to build it to whoever picks it up — unless a real constraint forces an approach.
- **Draw the scope line.** A short "not this" is worth a paragraph of prose; it's where scope creep dies.
- **Carry the context.** Links, examples, error text, screenshots, the person to ask. Everything you know that the reader doesn't.
- **One issue, one thing.** If the outcome has an "and" in it, it's probably two issues.

Match the repo's own template and labels. Short and complete beats long and hedged.
